# 深度分析场景数据来源分析

## 数据来源现状

### 当前可获取的数据

**YARN ResourceManager API**:
| 数据项 | 用途 | 场景适用性 |
|--------|------|-----------|
| allocated_vcores / allocated_memory_mb | 分配的资源总量 | 资源浪费检测 |
| running_containers | 容器数量 | Executor 状态 |
| diagnostics | 诊断信息（kill 原因） | Shuffle 失败、Executor 丢失 |
| state / final_status | 应用状态 | 任务卡住判断 |
| elapsed_time | 运行时长 | 执行时间对比 |

**Spark History Server API**:
| 数据项 | 用途 | 场景适用性 |
|--------|------|-----------|
| executor_count | Executor 数量 | 资源分析 |
| peak_executor_memory_mb | 峰值内存 | 资源浪费检测 |
| memory_spilled_mb | 内存溢出量 | OOM 分析 |
| shuffle_read_mb / shuffle_write_mb | Shuffle 数据量 | Shuffle 失败分析 |
| input_bytes_mb | 输入数据量 | 数据漂移 |
| Stage 信息（汇总） | Stage Shuffle metrics | 性能分析 |
| app_duration_ms | 应用执行时长 | 历史对比 |

## 各场景数据需求分析

### 场景1: 数据倾斜诊断

**判断**: ❌ **History + YARN 不够**

**当前能获取**:
- Stage 汇总信息（shuffle 数据量）
- Executor 数量

**缺失的关键数据**:
| 数据项 | 用途 | 来源 |
|--------|------|------|
| **Task 级别执行时间分布** | 判断是否倾斜（某个 Task 时间异常） | Spark History `/stages/{stageId}/taskSummary` |
| **Task 输入数据量分布** | 定位哪个 Task 数据量大 | Spark History `/stages/{stageId}/tasks` |
| **Partition ID** | 定位倾斜的 Partition | Task 详情 |
| **Key 分布统计** | 分析哪个 Key 导致倾斜 | ❌ Spark 不直接提供，需要代码层面加统计 |

**需要额外获取**:
```
Spark History API:
/api/v1/applications/{app_id}/stages/{stage_id}/taskSummary
  - 返回 Task 时间分布统计

/api/v1/applications/{app_id}/stages/{stage_id}/tasks
  - 返回每个 Task 的详细信息（duration, input_bytes）
```

**结论**: 需要调用 **Task 级别 API**，当前只获取了 Stage 汇总。

### 场景2: Shuffle 失败根因

**判断**: ⚠️ **部分足够，需要补充 Executor 日志**

**当前能获取**:
- YARN diagnostics（Container kill 原因）
- Executor 数量变化（通过历史对比）
- Shuffle 数据量

**缺失的关键数据**:
| 数据项 | 用途 | 来源 |
|--------|------|------|
| **Executor 丢失时间点** | 判断是哪个 Executor 先丢失 | YARN diagnostics（部分包含） |
| **Shuffle Service 状态** | 判断 Shuffle Service 是否正常 | ❌ 需要单独 API 或日志 |
| **Executor 日志** | 查看 Executor 端的具体错误 | ❌ YARN 只提供 diagnostics 概要 |

**需要额外获取**:
```
YARN Node Manager API（可选）:
/ws/v1/node/containers/{container_id}
  - 获取特定 Container 的详细日志

Shuffle Service 日志（可选）:
  - 外部 Shuffle Service 的状态检查
```

**结论**: YARN diagnostics 通常能判断根因（内存超限、磁盘满等），但复杂场景可能需要 Executor 日志。

```
Driver 日志分析:
  - 提取每行日志时间戳
  - 计算相邻日志时间差

Spark History API:
/api/v1/applications/{app_id}/stages?status=running
  - 获取正在运行的 Stage
```

**结论**: **足够**，需要增强 Driver 日志时间戳分析能力。

## Executor 日志获取方式

### Executor 日志来源分析

Executor 日志与 Driver 日志不同，存储在 YARN Container 中，需要额外获取。

**当前项目状态**:
- `yarn_log.py` 只实现了应用级别信息获取
- 测试文件中提到 `_fetch_container_log` 但实际未实现
- 需要补充 Container 级别日志获取能力

---

### 获取方式汇总

| 方式 | API/命令 | 适用场景 | 复杂度 |
|------|---------|---------|--------|
| **YARN Node Manager API** | `/ws/v1/cluster/apps/{app_id}/containers/{container_id}/logs` | YARN on Hadoop | ⭐⭐⭐ |
| **yarn logs 命令** | `yarn logs -applicationId {app_id}` | 命令行聚合所有日志 | ⭐⭐ |
| **Spark History Server** | `/api/v1/applications/{app_id}/executors` | Executor 信息（不含完整日志） | ⭐⭐ |
| **K8s Pod 日志** | `kubectl logs {pod_name}` 或 K8s API | Spark on K8s | ⭐⭐⭐ |

### 方式1: YARN Node Manager API（推荐）

**API 端点**:
```
GET /ws/v1/cluster/apps/{application_id}/containers
  → 获取 Container 列表（包含 container_id）

GET /ws/v1/cluster/apps/{application_id}/containers/{container_id}/logs
  → 获取特定 Container 的日志
```

**Container ID 格式**:
```
container_e01_{cluster_timestamp}_{app_seq}_{attempt_seq}_{container_seq}

示例: container_e01_1234567890123_0001_01_000002
  - e01: 集群 ID
  - 1234567890123: 集群启动时间戳
  - 0001: Application 序号
  - 01: Attempt 序号
  - 000002: Container 序号（000001 是 Driver，000002+ 是 Executor）
```

**实现流程**:
```
1. 获取应用信息 → YARN RM API
2. 获取 Container 列表 → /containers 端点
3. 过滤 Executor Container（container_seq > 1）
4. 获取每个 Container 日志 → /containers/{id}/logs
```

**完整 URL 示例**（通过 Knox Gateway）:
```
https://ali-odp-test-01.huan.tv:8443/gateway/default/yarn/ws/v1/cluster/apps/application_1234567890123_0001/containers/container_e01_1234567890123_0001_01_000002/logs
```

**返回内容**:
- stdout: 标准输出日志
- stderr: 标准错误日志

### 方式2: yarn logs 命令（简单聚合）

**命令**:
```bash
yarn logs -applicationId application_1234567890123_0001
```

**特点**:
- 聚合所有 Container 日志到一个输出
- 需要在有 yarn 命令的环境执行
- 输出量大，需要解析过滤

**Agent 调用方式**:
```python
result = subprocess.run(
    ["yarn", "logs", "-applicationId", app_id],
    capture_output=True,
    text=True
)
# 解析输出，按 Container 分割
```

```python
def get_containers(self, application_id: str) -> List[str]:
    """获取应用的所有 Container ID"""
    url = f"{self.gateway_url}/ws/v1/cluster/apps/{application_id}/containers"
    response = requests.get(url, auth=self.auth, timeout=15, verify=False)
    
    if response.status_code == 200:
        data = response.json()
        containers = data.get("containers", {}).get("container", [])
        return [c.get("id") for c in containers]
    return []

def fetch_container_log(self, application_id: str, container_id: str) -> str:
    """获取特定 Container 的日志"""
    url = f"{self.gateway_url}/ws/v1/cluster/apps/{application_id}/containers/{container_id}/logs"
    response = requests.get(url, auth=self.auth, timeout=30, verify=False)
    
    if response.status_code == 200:
        return response.text
    return ""

def fetch_executor_logs(self, application_id: str) -> Dict[str, str]:
    """获取所有 Executor 日志"""
    containers = self.get_containers(application_id)
    executor_logs = {}
    
    for container_id in containers:
        # Container 序号 > 1 的通常是 Executor
        seq = int(container_id.split("_")[-1])
        if seq > 1:  # Executor
            log = self.fetch_container_log(application_id, container_id)
            executor_logs[container_id] = log
    
    return executor_logs
```

### Executor 日志在深度分析中的应用

| 场景 | Executor 日志用途 | 是否必须 |
|------|------------------|---------|
| **场景2: Shuffle 失败** | 查看 Shuffle Service 连接错误、磁盘错误 | ⚠️ 可选（YARN diagnostics 通常足够） |
| **场景9: 任务卡住** | 确认 Executor 是否存活、心跳状态 | ⚠️ 可选（Driver 日志心跳通常足够） |
| **场景1: 数据倾斜** | 查看 Task 执行详情 | ❌ 不需要（Spark History Task API 更合适） |

**结论**: Executor 日志在深度分析中属于**可选补充**，YARN diagnostics 和 Driver 日志通常能覆盖主要场景。
