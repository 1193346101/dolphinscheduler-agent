# DolphinScheduler Agent 项目宣讲文稿

---

## 第一部分：项目背景与目标

### 1.1 为什么需要 Agent？

**痛点分析**：

| 场景 | 现状问题 | 解决目标 |
|------|----------|----------|
| 任务失败处理 | 运维人工查看日志 → 分析原因 → 手动执行修复命令 | 自动接收告警 → 智能分析 → 低风险自动修复，高风险发起审批 |
| 血缘查询 | 翻阅代码文档，人工梳理依赖关系 | 自动扫描 → 可视化展示依赖链路 → 隐式依赖分析 |
| 隐式依赖检查 | 手动检查 SUB_PROCESS/DEPENDENT 配置 | 自动检测表血缘驱动的缺失依赖 → 告警提醒 |

### 1.2 项目定位

**DolphinScheduler Agent** = 智能运维助手

- **告警自动化**: 接收 DolphinScheduler 失败告警，自动分析并修复
- **工作流血缘可视化**: 扫描工作流依赖关系，生成可视化图表
- **隐式依赖分析**: 检测 SUB_PROCESS、DEPENDENT、表血缘驱动的隐式依赖，识别缺失 DEPENDENT 任务

#### 3.1.1 DSCLIClient - DolphinScheduler 操作封装

**文件**: `src/integrations/dsctl_wrapper.py`

**设计决策**: 为什么封装 CLI 而不是直接调用 API？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接 API** | 灵活、响应快 | 需处理认证、版本兼容、参数组装、错误解析 |
| **CLI 封装** | 简单稳定、已封装好 | subprocess 调用略慢 |

**选择 CLI 的原因**:
1. `dsctl` 已处理 DolphinScheduler 3.x 的复杂认证（Token 获取、刷新）
2. 命令参数标准化，避免手写 JSON 请求体
3. CLI 自带错误提示，便于调试

**核心方法一览**：

| 方法 | 功能 | 状态 |
|------|------|------|
| `workflow_run()` | 运行工作流 | ⚠️ 缺少 worker_group/tenant 参数 |
| `workflow_instance_recover_failed()` | 恢复失败实例 | ✅ 正常 |
| `get_task_logs()` | 获取任务日志 | ✅ 正常 |
| `workflow_instance_export()` | 导出实例 YAML | ✅ 正常 |
| `workflow_instance_edit_patch()` | 编辑实例 | ✅ 正常 |

### 3.2 Skills 模块 - 错误分析专家系统

**目录**: `src/skills/`

#### 3.2.1 设计理念

**Skill = 快速预判器**

核心思想: 先用规则快速匹配已知错误，只有未知错误才调用 LLM

**分类体系**：

| Category | 含义 | 处理方式 | 示例 |
|----------|------|----------|------|
| **AUTO_FIXABLE** | 已知可自动修复 | 直接返回修复方案 | 分区缺失 → 检查OSS路径 |
| **RESOURCE_SUGGESTED** | 资源问题 | 智能计算配置 + LLM验证 | OOM → 增加executorMemory |
| **KNOWN_NEEDS_LLM** | 已知类型需推理 | 给 LLM 提上下文提示 | ClassNotFound → 提供类路径 |
| **UNKNOWN** | 未知错误 | 完全交给 LLM，记录候选 | 新错误类型 |

```python
def scan_project(self, project_code, project_name, ds_api_url, ds_api_token):
    # 1. 创建 DSCLI 客户端
    dsctl = DSCLIClient(api_url=ds_api_url, api_token=ds_api_token)
```

**_parse_workflow 详细流程**：

```python
def _parse_workflow(self, workflow, dsctl, graph, all_tables):
    workflow_code = workflow["code"]
    
    # 1. 获取工作流详细信息
    detail_result = dsctl.describe_workflow(project_code, workflow_code)
    
    # 2. 创建 WorkflowNode
    workflow_node = WorkflowNode(
        code=workflow_code,
        name=workflow_data["name"],
        schedule_type="CRON" if schedule_cron else "MANUAL",
        is_sub_workflow=False
    )
    graph.nodes.workflows.append(workflow_node)
    
    # 3. 解析任务定义
    for task in tasks_data:
        task_node = TaskNode(
            code=task["code"],
            name=task["name"],
            task_type=task["type"],
            workflow_code=workflow_code,
            spark_main_class=task.get("taskParams", {}).get("mainClass")
        )
        graph.nodes.tasks.append(task_node)
        
        # 4. 添加任务所属关系
        graph.edges.workflow_contains_task.append({source: workflow_code, target: task_code})
        
        # 5. 解析任务依赖
        for dep in task.get("dependence", []):
            graph.edges.task_depends_task.append({source: task_code, target: dep["code"]})
        
        # 6. 解析 SQL 提取表
        if task["type"] == "SPARK":
            sql = task.get("taskParams", {}).get("rawScript", "")
            tables = self.sql_parser.extract_tables(sql)
            for table in tables["input"]:
                graph.edges.task_consumes_table.append({source: task_code, target: table})
            for table in tables["output"]:
                graph.edges.task_produces_table.append({source: task_code, target: table})
```

#### 3.3.2 SQLParser - SQL 表名解析（CTE 过滤）

**文件**: `src/graph/sql_parser.py`

**CTE 过滤逻辑**：

```python
def _extract_cte_names(self, sql: str) -> List[str]:
    """提取 WITH 子句中的 CTE 名称"""
    cte_names = []
    
    # 找 WITH 子句
    with_match = re.search(r'WITH\s+', sql, re.IGNORECASE)
    if not with_match:
        return cte_names
    
    # 匹配所有 CTE 定义: cte_name AS (...)
    cte_def_pattern = r'([a-zA-Z_][a-zA-Z0-9_]*)\s+AS\s*\('
    
    for match in re.finditer(cte_def_pattern, sql[with_match.end():], re.IGNORECASE):
        cte_names.append(match.group(1).lower())
    
    return cte_names

def _extract_with_regex(self, sql: str) -> Dict:
    # 先提取 CTE 名称（用于排除）
    cte_names = self._extract_cte_names(sql)
    
    # 提取输入表（FROM/JOIN）
    for match in self._from_re.finditer(sql):
        table = match.group(1)
        # 排除 CTE 名称
        if table.lower() not in cte_names:
            result["input"].append(table)
```

#### 3.3.4 ImplicitDependencyAnalyzer - 隐式依赖分析（新增）

**文件**: `src/tools/implicit_dependency_analyzer.py`

**分析目标**：

| 依赖类型 | 检测方式 | 说明 |
|----------|----------|------|
| **SUB_PROCESS** | 任务类型为 SUB_PROCESS | 父工作流调用子工作流（显式） |
| **DEPENDENT** | 任务类型为 DEPENDENT | 等待外部工作流完成（显式） |
| **表血缘驱动** | Consumer 输入 = Producer 输出 | 隐式依赖（需告警） |

**分析逻辑**：

```python
def _analyze_table_lineage_dependencies(self, ...):
    """
    只检查：独立工作流使用其他工作流的输出表
    子工作流之间的依赖在父工作流 DAG 中已体现
    """
    
    # 构建工作流输入/输出表映射
    workflow_produces: Dict[int, Set[str]] = {}  # 工作流输出的表
    workflow_consumes: Dict[int, Set[str]] = {}  # 工作流输入的表
    
    # 过滤 CTE/临时视图名称
    real_shared_tables = {
        t for t in shared_tables
        if not t.startswith("view_") and not t.startswith("temp_")
        and not t.startswith("hdfs:")
    }
    
    # 只将独立工作流作为 consumer
    for consumer_wf_code in independent_workflow_codes:
        for producer_wf_code in workflow_produces:
            shared = workflow_consumes[consumer] & workflow_produces[producer]
            if shared:
                # 检查是否有 DEPENDENT 任务等待
                has_dependent = producer in dependent_waiting.get(consumer, set())
                if not has_dependent:
                    missing_deps.append(...)
```

**判断规则**：

| Consumer 类型 | Producer 类型 | 是否检查 DEPENDENT | 原因 |
|--------------|--------------|-------------------|------|
| 独立工作流 | 子工作流 | ✅ 检查 | 使用输出表无显式依赖 |
| 独立工作流 | 独立工作流 | ✅ 检查 | 跨工作流表依赖 |
| 子工作流 | 子工作流 | ❌ 不检查 | 父 DAG 中已有任务依赖 |

```python
            original_script = task['task_params']['rawScript']
            
            # 3. 应用修改
            modified_script = original_script.replace(wrong_cmd, correct_cmd)
            
            # 4. 修改实例 + 同步到定义
            edit_result = dsctl.workflow_instance_edit_task_script(
                process_instance_id,
                task_name,
                modified_script,
                sync_definition=True
            )
            
            # 5. 恢复失败任务
            recover_result = dsctl.workflow_instance_recover_failed(process_instance_id)
```

```python
    # recover-failed 只执行失败任务，不重跑成功任务
    return dsctl.workflow_instance_recover_failed(process_instance_id)
```

## 第五部分：关键技术决策

### 5.1 CLI 封装 vs 直接 API

**决策**: 使用 `dsctl` CLI 而非直接调用 DolphinScheduler REST API

**原因**: CLI 已处理复杂认证、版本兼容、参数组装

---

### 5.2 LangGraph 状态机

**决策**: 使用 LangGraph 编排告警处理流程

**原因**:
- 流程可视化：可导出流程图
- 条件分支：根据风险等级路由到不同处理节点
- 状态持久化：可暂停恢复长时间操作

---

### 5.3 Skill 规则优先

**决策**: 先用规则匹配（Skill），再 LLM Fallback

**原因**:
- 成本：已知错误用规则，不消耗 LLM token
- 速度：规则匹配毫秒级，LLM 调用秒级
- 准确性：已知错误规则更精准

---

### 5.4 隐式依赖只检查独立工作流

**决策**: 只将独立工作流作为表血缘分析的 consumer

**原因**:
- 子工作流在父工作流 DAG 中已被调度
- 子工作流之间的依赖在父 DAG 中体现
- 独立工作流使用子工作流输出表才需要 DEPENDENT 任务
