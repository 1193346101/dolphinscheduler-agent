# DolphinScheduler Agent 深度分析场景设计

## 文档说明

本文档记录 Agent 对复杂错误进行深度分析的场景设想。

**状态**: 🚧 设计中（场景尚未完整，持续补充）

**目标**: 简单错误（拼写、类名不存在等）在上线前测试就能发现，Agent 的核心价值在于分析**复杂耗时**的错误场景。

---

## 场景列表

### 一、数据倾斜/性能分析类（7个）

#### 场景1: 数据倾斜诊断

**现象**: Spark Stage 卡在某个 Task，其他 Task 快速完成

**人工分析难点**:
- 需要看 Spark UI 的 Task 分布图
- 判断是 Key 分布不均还是数据本身问题
- 需要分析具体哪个 Key 导致倾斜

**Agent 深度诊断价值**:
- 自动判断：是增加资源能解决，还是必须改代码（加盐、Broadcast Join）
- 定位倾斜的具体 Partition/Key
- 给出具体修改建议（而不是笼统的"建议优化代码")

**关键数据源**:
- Spark UI Task 分布（需要 API 或日志解析）
- Spark History Server metrics

**实现难度**: ⭐⭐⭐⭐（需要获取 Task 级别数据）

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 提取失败 Stage ID | Driver 日志正则匹配 `Stage \d+ failed` | stage_id |
| 2 | 获取该 Stage 的 Task 列表 | Spark History Server API `/api/v1/applications/{app_id}/stages/{stage_id}/taskSummary` | 各 Task 执行时间分布 |
| 3 | 计算 Task 时间离散度 | Agent 内部计算 | max_time / median_time 比值 |
| 4 | 判断倾斜类型 | 比值 > 3 = 数据倾斜；比值 < 1.5 = 非倾斜 | 倾斜判定结果 |
| 5 | 定位倾斜 Task | 找出执行时间最长的 Task | task_id, partition_id |
| 6 | 分析倾斜原因 | 查看该 Task 的输入数据量 | input_bytes 异常大？ |
| 7 | 生成建议 | 综合分析：数据量异常 → 建议加盐；广播表太大 → 建议 SMJ | 具体修复方案 |

**🔧 需要的工具/API**:

| 工具 | 用途 | 当前状态 |
|------|------|----------|
| Spark History Server API | 获取 Stage/Task 详情 | ✅ 已有调用能力 |
| Driver 日志解析 | 提取 Stage ID、错误信息 | ✅ 已有 |
| Task 时间分析脚本 | 计算离散度、定位倾斜 | ❌ 需新增 |

**📤 输出结果格式**:

```json
{
  "diagnosis_type": "data_skew",
  "stage_id": 15,
  "skew_ratio": 8.5,
  "skew_task": {
    "task_id": 123,
    "partition_id": 5,
    "duration_ms": 180000,
    "input_bytes": 524288000
  },
  "median_task_duration_ms": 21000,
  "root_cause": "partition_5 数据量异常大（占比 42%）",
  "suggestion": {
    "type": "code_change",
    "method": "salting",
    "description": "对 Join Key 加盐分散数据",
    "alternative": "改用 BroadcastHashJoin（如果小表 < 200MB）"
  }
}
```

#### 场景7: 执行计划异常

**现象**: Spark SQL 执行效率突然下降，但日志没有报错

**人工分析难点**:
- 需要看 Spark UI 的执行计划图
- 判断 Spark 是否选择了错误的 Join 策略（本该 Broadcast Join 却用了 SortMergeJoin）
- 判断是否分区裁剪失效（本该只读某个分区却读全表）

**Agent 深度诊断价值**:
- 从日志提取执行计划关键信息
- 对比历史执行时间，识别异常
- 定位：是数据量变化导致执行计划切换，还是统计信息缺失

**关键数据源**:
- Spark 日志中的执行计划信息
- Spark History Server 查询计划

**实现难度**: ⭐⭐⭐

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 按需拉取历史一周日志 | dsctl get_task_logs（成功实例） | 历史 Stage 执行时间列表 |
| 2 | 建立历史基线 | 计算各 Stage 平均执行时间 | baseline_times |
| 3 | 提取当前执行时间 | Driver 日志 `Stage \d+ finished in \d+ ms` | current_times |
| 4 | 对比识别异常 Stage | 当前时间 > 历史 * 2 = 异常 | 异常 Stage 列表 |
| 5 | 提取执行计划信息 | Driver 日志 `Join strategy: .*` 或 Spark History | Join 类型、扫描分区数 |
| 6 | 分析异常原因 | 对比历史 Join 策略、分区数 | 策略切换？分区裁剪失效？ |
| 7 | 生成建议 | 综合分析结果 | 具体修复方案 |

**🔧 需要的工具/API**:

| 工具 | 用途 | 当前状态 |
|------|------|----------|
| dsctl get_task_logs | 拉取历史日志 | ✅ 已有 |
| Driver 日志解析 | 提取 Stage 时间、Join 策略 | ⚠️ 需增强 |
| 执行计划对比脚本 | 分析策略变化 | ❌ 需新增 |
| Spark History Server API | 获取 SQL 执行计划 | ✅ 已有调用能力 |

**📤 输出结果格式**:

```json
{
  "diagnosis_type": "execution_plan_change",
  "abnormal_stages": [
    {
      "stage_id": 3,
      "current_duration_ms": 120000,
      "baseline_duration_ms": 25000,
      "ratio": 4.8
    }
  ],
  "root_cause": "Join 策略从 BroadcastHashJoin 切换到 SortMergeJoin",
  "detail": {
    "previous_strategy": "BroadcastHashJoin",
    "current_strategy": "SortMergeJoin",
    "table_size_change": "广播表从 8MB 增长到 15MB，超过阈值 10MB"

  "suggestion": {
    "type": "config_change",
    "config": "spark.sql.autoBroadcastJoinThreshold",
    "value": "20MB",
```

#### 场景9: 任务卡住不报错

**现象**: 任务运行几小时不动，没有 ERROR 日志，但某个 Stage 一直不推进

**人工分析难点**:
- 日志没有明显错误，只是看不到进度
- 需要看 Spark UI 判断是哪个 Stage 卡住
- 判断是数据倾斜、网络问题、还是 Driver hang 了

**Agent 深度诊断价值**:
- 分析日志时间戳，检测"长时间无输出"
- 结合 YARN ResourceManager 查看 Executor 状态
- 区分：数据倾斜（某个 Task 卡）vs Driver hang（整个任务无响应）

**关键数据源**:
- Driver 日志时间戳
- YARN ResourceManager Executor 状态

**实现难度**: ⭐⭐⭐

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 分析日志时间戳间隔 | Driver 日志每行时间戳 | 各时间段的日志输出频率 |
| 2 | 检测长时间无输出 | 计算相邻日志时间差 > 5分钟 = 异常 | 无输出时间段列表 |
| 3 | 判断卡住类型 | 无输出但有心跳 = Task 卡；无心跳 = Driver hang | 卡住类型 |
| 4 | 获取 Executor 状态 | YARN RM API `/ws/v1/cluster/apps/{app_id}` | Executor 数量、状态 |
| 5 | 检查 Executor 是否丢失 | 对比配置 Executor 数 vs 实际存活数 | Executor 丢失情况 |
| 6 | 检查心跳日志 | Driver 日志 `Executor heartbeat` | 心跳是否正常 |
| 7 | 综合判断并生成建议 | 多源数据综合分析 | 根因 + 建议 |

**🔧 需要的工具/API**:

| 工具 | 用途 | 当前状态 |
|------|------|----------|
| Driver 日志时间戳分析 | 检测无输出时段 | ❌ 需新增 |
| YARN ResourceManager API | 获取 Executor 状态 | ✅ 已有 |
| 心跳检测脚本 | 判断 Driver/Executor 状态 | ❌ 需新增 |

**📤 输出结果格式**:

```json
{
  "diagnosis_type": "task_hang",
  "hang_type": "executor_lost",
  "silent_periods": [
    {
      "start_time": "14:30:00",
      "end_time": "14:55:00",
      "duration_minutes": 25
    }
  ],
  "executor_status": {
    "configured": 4,
    "alive": 2,
    "lost_executors": [
      {"id": 1, "lost_time": "14:32:00"},
      {"id": 3, "lost_time": "14:35:00"}
    ]
  },
  "root_cause": "Executor 因内存不足被 YARN kill",
  "suggestion": {
    "type": "resource_change",
    "executor_memory": "4g → 6g",
    "reason": "Executor 内存不足以处理当前数据量"
  }
}
```

#### 场景10: 资源配置浪费

**现象**: 任务配置了 8g Executor 内存，但实际峰值只用了 2g

**人工分析难点**:
- 任务正常执行，没人发现问题
- 需要看 Spark History Server 的内存使用曲线
- 这是"隐性浪费"，不会主动排查

**Agent 深度诊断价值**:
- 从 History Server 获取峰值内存使用
- 对比配置内存，识别浪费比例
- 建议：降低资源配置，释放集群资源给其他任务

**关键数据源**:
- Spark History Server 内存 metrics

**实现难度**: ⭐⭐（已有获取 metrics 能力）

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 获取当前配置 | Driver 日志 config_lines 解析 | executor_memory 配置值 |
| 2 | 获取峰值内存使用 | Spark History API Executor 内存 metrics | peak_memory_used |
| 3 | 计算使用率 | peak_memory / configured_memory | 使用率百分比 |
| 4 | 判断是否浪费 | 使用率 < 50% = 浪费 | 判定结果 |
| 5 | 计算建议配置 | peak_memory * 1.5（留余量） | 建议内存值 |
| 6 | 生成优化建议 | 综合分析 | 降配建议 |

**🔧 需要的工具/API**:

| 工具 | 用途 | 当前状态 |
|------|------|----------|
| Spark History Server API | 获取 Executor 内存 metrics | ✅ 已有 |
| Driver 日志解析 | 提取配置信息 | ✅ 已有 |
| 内存对比脚本 | 计算使用率 | ❌ 需新增 |

**📤 输出结果格式**:

```json

  },

    "type": "config_change",
    "executor_memory": "3g",

  }
```

#### 场景11: 广播变量过大

**现象**: BroadcastHashJoin 失败或超时

**人工分析难点**:
- 需要判断广播的数据量是否超过阈值（默认 10MB）
- 判断是数据量突然变大，还是本来就不该用 Broadcast Join

**Agent 深度诊断价值**:
- 从日志提取广播数据量
- 判断是否超过阈值
- 建议：改用 SortMergeJoin 或调大 `spark.sql.autoBroadcastJoinThreshold`

**关键数据源**:
- Spark 日志中的广播大小信息

**实现难度**: ⭐⭐

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 提取广播失败信息 | Driver 日志 `Broadcast.*failed|Could not broadcast` | 失败确认 |
| 2 | 提取广播数据量 | Driver 日志 `Broadcast size: \d+ bytes` 或 `relation size: \d+` | broadcast_size_bytes |
| 3 | 获取当前阈值配置 | Driver 日志 `spark.sql.autoBroadcastJoinThreshold` | threshold_config |
| 4 | 判断是否超阈值 | broadcast_size > threshold | 超阈值判定 |
| 5 | 分析表大小变化 | 按需拉取历史日志对比表大小 | 表是否突然变大 |
| 6 | 生成建议 | 超阈值 → 调大阈值或改 SMJ | 具体方案 |

**🔧 需要的工具/API**:

### 二、多源日志综合分析类（2个）

#### 场景2: Shuffle 失败根因分析

**现象**: FetchFailedException 或 Shuffle 连接失败

**人工分析难点**:
- Shuffle 失败可能是多种原因：网络、磁盘、内存、Executor 丢失
- 需要查多个地方：Shuffle Service 日志、Executor 日志、YARN 日志
- 判断是临时问题（重跑即可）还是持久问题（需要加资源）

**Agent 深度诊断价值**:
- 综合分析多个日志源，定位根因
- 区分：临时问题（建议重跑）vs 持久问题（需调整配置）

**关键数据源**:
- Driver 日志 Shuffle 错误信息
- Executor 日志
- YARN ResourceManager diagnostics

**实现难度**: ⭐⭐⭐⭐（多源日志综合）

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 提取 Shuffle 失败信息 | Driver 日志 `FetchFailedException` | 失败确认、shuffle_server 地址 |
| 2 | 提取失败类型 | Driver 日志具体错误信息 | `connection refused` / `timeout` / `disk error` |
| 3 | 获取 YARN diagnostics | YARN RM API `/ws/v1/cluster/apps/{app_id}` | diagnostics 文本 |
| 4 | 分析 diagnostics 关键词 | 正则匹配关键错误词 | 根因类型（内存/磁盘/网络） |
| 5 | 检查 Executor 状态 | YARN API Executor 列表 | Executor 丢失情况 |
| 6 | 检查 Shuffle Service | 尝试连接 Shuffle Service（可选） | Shuffle Service 是否存活 |
| 7 | 综合判断根因 | 多源数据汇总分析 | 根因 + 临时/持久判定 |

**🔧 需要的工具/API**:

| 工具 | 用途 | 当前状态 |

#### 场景3: 文件块处理失败

**现象**: 读取 OSS/HDFS 文件时，某个文件块失败

**人工分析难点**:
- 日志可能只显示"File corrupted"，但不知道哪个文件
- 需要手动排查：哪个 Partition、哪个文件路径
- 判断是文件真的损坏还是临时读取失败

**Agent 深度诊断价值**:
- 提取失败的具体文件路径
- 使用 ossutil 验证文件是否损坏
- 区分：文件损坏（需修复上游）vs 临时问题（重跑）

**关键数据源**:
- Driver 日志失败路径
- ossutil 验证结果

**实现难度**: ⭐⭐（已有 ossutil 验证能力）

---

**📋 详细排查过程**:

| 步骤 | 操作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 提取失败文件路径 | Driver 日志 `oss://.*|hdfs://.*` + `corrupt|failed|error` | failed_path |
| 2 | 提取失败 Partition | Driver 日志 `Partition \d+` | partition_id |
| 3 | 使用 ossutil 验证 | ossutil ls {failed_path} | 文件是否存在、大小 |
| 4 | 验证文件完整性 | ossutil ls -l 查看文件详情 | 文件大小是否正常 |
| 5 | 判断失败类型 | 文件不存在 → 数据缺失；文件存在但异常 → 损坏；文件正常 → 临时读取失败 | 失败类型 |
